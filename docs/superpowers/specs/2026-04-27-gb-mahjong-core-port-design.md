# GB-Mahjong-JS 核心迁移设计

## 背景

当前仓库基本仍处于脚手架状态，现有 `lib/index.js` 与 `lib/__tests__/gbMahjongJs.test.js` 尚未承载国标麻将核心能力。
真实业务逻辑位于 `third_party/GB-Mahjong`，为一个可运行、带有较完整单测样例的 C++ 实现。

本次工作的目标不是重新设计一套新规则库，而是把 `third_party/GB-Mahjong` 的核心能力迁移为 Node.js 版本，并建立可以持续验证一致性的测试体系。

## 目标

- 迁移 C++ 版本中的核心麻将领域模型与规则计算能力。
- 在 JavaScript 中保持与 C++ 版本尽可能一致的输入语义、输出语义与错误码语义。
- 将 C++ `unit_test.cpp` 中可在 Node/Jest 中表达的测试样例完整迁移为 Jest 测试。
- 为后续发布更友好的 JS API 保留空间，但第一阶段以“行为一致”优先。

## 非目标

- 不在第一阶段重构为完全不同的 JS 风格领域模型。
- 不在第一阶段引入 UI、CLI、浏览器适配或网络接口。
- 不以性能优化为首要目标；先保证正确性与可验证性。
- 不修改 `third_party/GB-Mahjong` 作为规则来源。

## 源实现分析

C++ 源实现可以分为四个主要部分：

- `tile.*`
  - 定义牌 ID、花色、点数、字符映射、位图常量以及 `Tile` 行为。
- `pack.*`
  - 定义顺子、刻子、杠、将、组合龙等牌组抽象与供牌信息。
- `handtiles.*`
  - 定义整手牌的数据结构、字符串解析、标准化输出、场况/和牌方式状态。
- `fan.*`
  - 定义和牌判定、听牌计算、番种枚举、排它逻辑、总番数选择与最终番表结果。

另外，`unit_test.cpp` 已经提供了迁移验收的基准样例，覆盖：

- 字符串解析与标准化输出
- 非法输入与错误码
- 听牌计算
- 和牌与算番结果
- 规则边界与多番种组合

## 迁移原则

### 1. 先兼容，再抽象

第一阶段内部 API 尽量贴近 C++ 实现，使迁移更直接、验证更容易。等核心行为稳定后，再考虑增加更符合 JS 使用习惯的包装层。

### 2. 先数据模型，再算法

任何番种与和牌逻辑都依赖统一、稳定的牌表示与字符串协议。因此必须先完成 `Tile`、`Pack`、`Handtiles` 的语义迁移，再迁移 `Fan`。

### 3. 测试与迁移同步推进

每个模块迁移时都要同步建立测试，不接受“先抄完算法再统一补测试”的做法。C++ 样例直接作为行为基准。

### 4. 保持错误语义稳定

`StringToHandtiles` 等接口返回的错误码在现有测试中已经承担协议语义。JS 版应保持相同错误码，避免后续定位成本上升。

## 目标架构

建议在 `lib/` 下建立明确的内核分层。

### `lib/core/tile.js`

职责：

- 维护牌 ID 常量、花色/点数映射、字符映射、位图常量。
- 提供 `Tile` 类及基础方法：比较、前驱后继、花色判断、字符输出、抓牌标记等。

设计约束：

- 保留与 C++ 相同的 tile id 编码，避免番种算法中大量位图逻辑失真。
- 尽量避免把数牌和字牌拆成不同对象类型，统一使用单一 `Tile` 实体。

### `lib/core/pack.js`

职责：

- 定义 `Pack` 类与牌组类型常量。
- 保持供牌者/吃碰杠来源的编码语义。
- 提供牌组展开、类型判断、组合龙位图访问等能力。

设计约束：

- `offer` 语义保持与 C++ 一致，因为它直接影响门清、明暗刻、和牌来源等判断。

### `lib/core/handtiles.js`

职责：

- 定义整副手牌对象。
- 管理副露、立牌、花牌与数量表。
- 提供字符串解析与标准化输出。
- 管理圈风、门风、自摸、绝张、海底、杠等状态。

设计约束：

- 解析结果必须可逆，即标准化输出与 C++ 规则一致。
- 内部保留“最后一张立牌”的语义，因为和牌/自摸/铳和依赖该状态。

### `lib/core/fan.js`

职责：

- 迁移 `fan_t`、番数表、番名表。
- 提供 `JudgeHu`、`JudgeHuTile`、`CalcTing`、`CountFan`。
- 保持番表、排它表、最大番数组合选择逻辑。

设计约束：

- 第一阶段不主动改写核心递归/搜索策略，只在 JS 中做等价表达。
- 若需要局部重构，仅允许为可读性或 JS 数据结构适配服务，不改变决策顺序与语义。
- `Fan` 在第一阶段保持为类实例接口，而不是静态函数集合。目标调用方式与 C++ 对齐，例如：

```js
const fan = new Fan();
fan.CountFan(handtiles);
const ting = fan.CalcTing(handtiles);
```

### `lib/index.js`

职责：

- 统一导出基础类型、常量与核心能力。
- 对外暴露兼容内核接口。

第一阶段建议导出：

- `Tile`
- `Pack`
- `Handtiles`
- `Fan`
- 关键常量与番种枚举

## 测试架构

### 测试分层

建议把测试拆分为四类：

- `lib/__tests__/tile.test.js`
  - 基础常量、字符映射、比较与标记行为。
- `lib/__tests__/handtiles.parse.test.js`
  - 字符串解析、标准化输出、错误码。
- `lib/__tests__/fan.hu-ting.test.js`
  - 和牌判断与听牌计算。
- `lib/__tests__/fan.count.test.js`
  - 番种组合、总番数、番表结果。

### 测试来源

测试样例优先来自 `third_party/GB-Mahjong/unit_test.cpp`，按以下顺序迁移：

1. 解析成功样例
2. 解析失败样例
3. 听牌样例
4. 代表性算番样例
5. 大番与复杂排它样例

### 测试范围

本阶段验收口径明确如下：

- 目标是把 `third_party/GB-Mahjong/unit_test.cpp` 中可在 Node/Jest 中表达的测试样例完整迁移，而不是只迁“代表性子集”。
- 若某些样例仅依赖 C++ 控制台打印格式，而不涉及核心规则语义，则允许在 Jest 中改写为等价的结构化断言。
- “完整测试”指完整覆盖其规则输入与结果断言，不要求逐字复刻 C++ 控制台输出文本。
- 若在迁移过程中发现上游测试本身存在歧义或注释与行为不一致，应在 JS 侧保留与 C++ 实际行为一致的断言，并在测试注释中说明。

### 测试断言策略

- 对解析：断言返回码、标准化字符串。
- 对听牌：断言听牌列表完全一致。
- 对和牌：断言 `JudgeHu` / `JudgeHuTile` 结果一致。
- 对算番：至少断言总番数与番种集合一致；在可行范围内断言每种番对应的 pack 结果。

### 验证原则

以 C++ 行为作为事实标准。若 JS 实现与直觉冲突但与 C++ 样例一致，应优先信任源实现，除非发现源实现或测试本身存在明确错误。

## 并行实施方案

用户要求使用 sub-agent 分模块迁移，因此实现阶段采用如下分工：

- Worker 1：`tile.js`、`pack.js`、相关基础测试
- Worker 2：`handtiles.js`、解析/标准化测试
- Worker 3：`fan.js` 中的和牌判断、特殊和型判断、`JudgeHu` / `JudgeHuTile` / `CalcTing` 与对应测试
- Worker 4：`fan.js` 中的 `CountFan`、番表排它逻辑、最大番数组合选择，以及全部番种结果测试迁移
- 主线程：统一目录结构、接口整合、冲突处理、最终验证

分工原则：

- 每个 worker 拥有清晰写入边界，减少冲突。
- 所有跨模块共享常量由主线程统一整理，避免重复定义。
- `fan.js` 由两个 worker 在职责上拆分，但最终整合由主线程完成。

## 风险与应对

### 风险 1：C++ 逻辑高度耦合，JS 直译后可读性差

应对：

- 第一轮先保真迁移。
- 仅在单元边界处提炼辅助函数，不做大规模结构重写。

### 风险 2：`unit_test.cpp` 规模较大，完整迁移易漏项

应对：

- 先迁最小闭环样例以建立可运行内核。
- 在同一实现周期内继续按类别补齐剩余样例，直到 `unit_test.cpp` 完整迁移完成。
- 尽量把测试数据结构化，避免重复手写断言模板。

### 风险 3：JS 数组/对象语义与 C++ 容器不同

应对：

- 用显式类和受控辅助函数代替松散对象。
- 数量表统一封装访问，避免 `undefined` 与 `0` 语义混淆。

### 风险 4：并行改动集中在 `fan.js`

应对：

- 由主线程先定义文件骨架与导出接口。
- Worker 3 和 Worker 4 分别负责独立方法区块与独立测试文件，降低合并成本。

## 交付定义

完成标准如下：

- Node 侧可创建 `Tile`、`Pack`、`Handtiles`、`Fan` 实例。
- 能解析并标准化输出 C++ 规则字符串。
- 能完成基础和牌判断与听牌计算。
- 能完成核心番种统计，并完整迁移 `third_party/GB-Mahjong/unit_test.cpp` 中可结构化表达的测试场景。
- Jest 测试可运行，并能稳定验证迁移行为。
- `README.md` 至少更新为可反映实际 API 的最低可用说明。

## 实现顺序

1. 建立目录结构与导出边界。
2. 迁移 `tile` / `pack`。
3. 迁移 `handtiles` 与解析协议。
4. 迁移 `JudgeHu` / `CalcTing`。
5. 迁移 `CountFan`。
6. 批量补足 `unit_test.cpp` 对应测试。
7. 统一跑 `eslint` 与 `jest` 验证。

## 结论

本次迁移采用“兼容内核优先、分层并行迁移、以 C++ 测试为准绳”的方案。这样可以在控制风险的前提下，把一个规则复杂的 C++ 麻将库可靠地转换为 Node.js 可维护实现，并为后续 API 友好化与发布打下基础。
